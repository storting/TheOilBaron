Git основы

main
↓ от
develop
↓ от
feature/TASK-xxx

1. Настройка GitHub Desktop

  • Скачайте с desktop.github.com
  • Войдите в свой GitHub аккаунт
  • Настройте Git: File → Options → Git

2. Клонирование репозитория

  • File → Clone repository
  • Выберите наш репозиторий
  • Укажите локальную папку
  • Нажмите Clone

3. Создание ветки

  • Нажмите Current branch → New branch
  • Название: feature/TASK-123-description
  • Убедитесь, что создаете от develop
  • Нажмите Create branch

4. Работа с изменениями

  • Левый столбец — измененные файлы
  • Ставим галочки у файлов для коммита
  • Пишем сообщение коммита внизу
  • Нажимаем Commit to feature/TASK-123
  • Publish branch (первый пуш)

5. Создание Pull Request

  • Branch → Create Pull Request
  • Откроется GitHub в браузере
  • Заполните шаблон PR
  • Назначьте ревьюверов

GitHub Desktop подробно

Визуальное руководство по GitHub Desktop

Основные элементы интерфейса

  • Current repository — выбор репозитория
  • Current branch — переключение веток
  • Changes — список измененных файлов
  • History — история коммитов
  • Fetch origin — обновить с сервера
  • Push/Pull — отправить/получить изменения

Горячие клавиши

  • Cmd/Ctrl + N — новый репозиторий
  • Cmd/Ctrl + O — открыть репозиторий
  • Cmd/Ctrl + Shift + N — новая ветка
  • Cmd/Ctrl + Enter — коммит
  • Cmd/Ctrl + P — пуш

Разрешение конфликтов в GitHub Desktop

  1. При возникновении конфликта появится уведомление
  2. Нажмите "Open in Visual Studio Code" (или другой редактор)
  3. Решите конфликт в редакторе (выберите нужные изменения)
  4. Сохраните файл и вернитесь в GitHub Desktop
  5. Нажмите "Commit merge"

Просмотр истории и откат изменений

  • Перейдите на вкладку "History"
  • Правый клик на коммите → Revert this commit
  • Или используйте "Undo" для последнего коммита
  • Для сброса изменений: Repository → Discard changes

Советы по GitHub Desktop

1. Всегда проверяйте Changes перед коммитом — что именно вы коммитите

2. Используйте встроенный редактор для написания хороших сообщений коммитов

3. Настройте внешний редактор кода в Preferences → Integrations

4. Включайте "Keep my local changes" при переключении веток, если не хотите терять изменения

Работа с конфликтными файлами

  • Сцены (*.unity) — разрешать вручную или UnityYAMLMerge
  • Префабы (*.prefab) — аналогично сценам
  • Настройки проекта (ProjectSettings/*) — согласовать с командой
  • Мета-файлы — обычно принимаем изменения с обеих сторон

Инструменты для Unity + Git

1. UnityYAMLMerge — для слияния сцен и префабов (встроен в Unity)

2. Git LFS — для больших файлов (текстуры, аудио, видео)

3. GitHub for Unity (устаревший) — лучше использовать GitHub Desktop

4. Plastic SCM — альтернатива Git, встроена в Unity

Чек-листы

Перед коммитом

  • Проект собирается без ошибок
  • Нет Debug.Log() в продакшен коде
  • Сообщение коммита по стандарту
  • Не добавились лишние файлы
  • Изменения соответствуют задаче

Еженедельный чек-лист

  • Обновить локальные ветки
  • Проверить нет ли конфликтов с develop
  • Удалить старые мерженые ветки
  • Сделать backup важных изменений
